# Offline Packages — Air-Gapped Deployment Guide

This directory holds all binary artifacts required to deploy the observability
stack on a **network-disconnected (air-gapped)** environment. Nothing in this
directory is tracked by git (see `.gitignore`) — you must populate it by
running `scripts/offline-prep.sh` on an internet-connected machine.

---


## Step 1 — Prepare packages on an internet-connected machine

```bash
# Clone or copy the repository to any machine with internet access
git clone https://github.com/scohmer/observability-dashboard.git
cd observability-dashboard

# Run the preparation script (requires: docker, curl, unzip)
bash scripts/offline-prep.sh

# Optionally pin specific versions:
NODE_EXPORTER_VERSION=1.8.2 \
WINDOWS_EXPORTER_VERSION=0.29.2 \
PROMTAIL_VERSION=3.0.0 \
bash scripts/offline-prep.sh
```

The script downloads approximately 1–2 GB of artifacts. It is safe to re-run
— already-downloaded files are skipped.

---

## Step 2 — Transfer packages to the air-gapped network

Transfer the **entire project directory** (or at minimum `offline-packages/`)
to the monitoring server in the air-gapped network. Recommended methods:

| Method | Notes |
|--------|-------|
| Encrypted USB drive | Most common; use VeraCrypt or BitLocker |
| Secure FTP / SFTP to a DMZ host | Works if there is a controlled transfer point |
| CD/DVD | For very restricted environments |
| Removable hard drive | For large deployments (multiple architectures) |

**Verify integrity after transfer** using the checksums in `MANIFEST.txt`:

```bash
# On the air-gapped monitoring server
cd offline-packages

# Verify Docker images
sha256sum -c <(grep "\.tar\.gz" MANIFEST.txt) 2>/dev/null | grep -v "^$"

# Or verify individual files:
sha256sum docker-images/grafana-grafana-latest.tar.gz
# Compare output against MANIFEST.txt
```

## Step 3 — Load Docker images on the monitoring server

```bash
# Requires: Docker installed and running
bash scripts/load-images.sh

# Preview without loading:
bash scripts/load-images.sh --dry-run

# Custom directory:
bash scripts/load-images.sh --images-dir /mnt/usb/offline-packages/docker-images
```

Verify images loaded correctly:

```bash
docker images | grep -E "grafana|prometheus|loki|promtail"
```

Expected output:
```
grafana/grafana    latest    ...
prom/prometheus    latest    ...
grafana/loki       latest    ...
grafana/promtail   latest    ...
```

## Step 4 — Run setup and start the stack

```bash
# Interactive setup — generates all configs
bash setup.sh

# Start the monitoring stack
docker compose up -d

# Check status
docker compose ps
docker compose logs -f
```

## Step 5 — Deploy agents to monitored hosts

The `offline-packages/` directory is referenced by all Ansible roles via the
`offline_packages_dir` variable in `ansible/group_vars/all.yml`.

```bash
cd ansible

# Deploy node_exporter (Linux and windows_exporter to Windows)
ansible-playbook playbooks/deploy-node-exporter.yml

# Deploy Promtail log shipper
ansible-playbook playbooks/deploy-promtail.yml

# Deploy to specific hosts only
ansible-playbook playbooks/deploy-node-exporter.yml --limit webserver01

# Check connectivity first
ansible all -m ping
```

### Linux hosts

The Ansible role copies binaries from `offline-packages/node_exporter/` and
`offline-packages/promtail/` directly to the target host using the `copy`
module — no internet access required on Linux targets.

### Windows hosts

The Ansible role copies the MSI and exe files from
`offline-packages/windows_exporter/` and `offline-packages/promtail/` to the
Windows target over WinRM — no internet access required on Windows targets.

## WinRM Setup on Windows Targets

Before Ansible can manage Windows hosts, WinRM must be configured. Run this
in an **elevated PowerShell** on each Windows target:

```powershell
# Enable WinRM
winrm quickconfig -q

# Allow NTLM authentication
winrm set winrm/config/service/auth '@{Negotiate="true"}'

# Allow HTTP (use HTTPS in production for security)
winrm set winrm/config/service '@{AllowUnencrypted="true"}'
Set-Item -Path WSMan:\localhost\Service\AllowUnencrypted -Value $true

# Set execution policy
Set-ExecutionPolicy RemoteSigned -Scope LocalMachine -Force

# Restart WinRM
Restart-Service WinRM

# For workgroup machines, trust the Ansible control node:
Set-Item WSMan:\localhost\Client\TrustedHosts -Value "MONITORING_SERVER_IP"
```

## Troubleshooting

### Docker image load fails

```bash
# Check available disk space
df -h

# Try loading manually
docker load < offline-packages/docker-images/grafana-grafana-latest.tar.gz

# Check image is not corrupted
file offline-packages/docker-images/grafana-grafana-latest.tar.gz
```

### node_exporter won't start after Ansible deployment

```bash
# On the target Linux host
systemctl status node_exporter
journalctl -u node_exporter -n 50
/usr/local/bin/node_exporter --version
```

### windows_exporter installation fails

```powershell
# Check MSI install log
Get-WinEvent -LogName Application | Where-Object {$_.Source -eq "MsiInstaller"} | Select-Object -First 10

# Verify MSI integrity
Get-FileHash C:\Windows\Temp\windows_exporter.msi -Algorithm SHA256
```

### Prometheus cannot scrape targets

```bash
# Check Prometheus targets page
curl http://localhost:9090/api/v1/targets | python3 -m json.tool | grep -A5 '"health"'

# Verify connectivity from monitoring server to target
curl http://TARGET_IP:9100/metrics | head -20
curl http://TARGET_IP:9182/metrics | head -20
```

### Grafana datasource fails

```bash
# Test Prometheus from inside Grafana container
docker exec grafana curl -s http://prometheus:9090/-/healthy

# Test Loki from inside Grafana container
docker exec grafana curl -s http://loki:3100/ready
```
